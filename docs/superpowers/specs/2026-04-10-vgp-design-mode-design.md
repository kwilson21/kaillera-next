# VGP Design Mode — Design Spec

## Problem

The virtual gamepad layout (`web/static/virtual-gamepad.js`) uses a responsive CSS system with `clamp()`, `svh` units, CSS custom properties, and 6+ media query breakpoints across portrait, landscape, narrow, short, and tablet variants. Adjusting this layout requires:

1. Editing CSS values blindly
2. Running a 100-device Playwright screenshot test
3. Reviewing static screenshots
4. Repeating until it looks right

This loop is too slow and disconnected. The developer needs to see changes across all devices simultaneously and manipulate elements directly.

## Solution

A single static HTML file (`tests/design-mode.html`) that renders the real VGP inside iframes at every target device's exact viewport dimensions. The developer selects a reference device, drags elements to reposition them, and sees the effect propagate across all viewports live. Changes are exported as JSON for Claude to translate into responsive CSS.

## Architecture

### Files

- `tests/design-mode.html` — the design tool (single file, no dependencies beyond vgp-test.html)
- `tests/vgp-test.html` — existing test page loaded inside each iframe (unchanged)
- `tests/vgp-devices.js` — shared device list (single source of truth for both design-mode and vgp-device-test)
- `tests/vgp-design-changes.json` — exported change log (generated by the tool)

### Shared Device List

Both `design-mode.html` and `vgp-device-test.mjs` import the device list from `tests/vgp-devices.js`. This eliminates drift between the design tool and the validation test. The file exports an array of `[name, chromeReduction, category]` tuples plus the chrome constants, matching the existing format in `vgp-device-test.mjs`.

### Serving

Serve from the repo root: `python3 -m http.server 18888` (from `/Users/kazon/kaillera-next`).

`vgp-test.html` references `/static/play.css` and `/static/virtual-gamepad.js`. When served from repo root, these paths resolve to `./static/` which doesn't exist at root — they live in `web/static/`. To fix this without modifying `vgp-test.html`, create a symlink: `ln -sf web/static static` at the repo root. This makes `/static/` resolve correctly whether served from root or `web/`.

Both `tests/design-mode.html` and `tests/vgp-test.html` are then accessible at `/tests/...`.

## Viewport Rendering

Each device viewport is an `<iframe>` at the real device pixel dimensions:

- Portrait: `width = device.viewport.width`, `height = device.viewport.height - chromeReduction`
- Landscape: `width = device.viewport.height`, `height = device.viewport.width - chromeReduction`
- Minimum: 300x200

Note: chrome reduction in landscape is an approximation. Real mobile browsers may auto-hide chrome in landscape. The existing Playwright test uses this same approximation, and designing for the reduced height is conservative (controls that fit in less space will fit in more).

The iframe loads `vgp-test.html` which includes the real `play.css` and `virtual-gamepad.js`.

The iframe is placed inside a container that applies `transform: scale(factor)` to fit the grid. The iframe itself stays at real dimensions so all CSS media queries, `svh` units, and `clamp()` values fire exactly as they would on the real device. The scale factor is computed from the zoom level and available grid space.

## Iframe Performance Strategy

~74 iframes is heavy. Mitigations:

1. **Lazy loading** — iframes outside the visible scroll area are created with `loading="lazy"` and `src` set only when they enter the viewport (via `IntersectionObserver` with a 200px root margin).
2. **Inert non-reference iframes** — only the reference iframe has active event listeners. All others have `pointer-events: none` on their containers.
3. **Deferred JS** — non-reference iframes load `vgp-test.html` which runs `VirtualGamepad.init()`. The DOM and CSS render correctly for visual preview even if JS hasn't fully initialized.
4. **Fallback** — if performance is unacceptable on a given machine, a "Lite Mode" toggle in the toolbar reduces to ~20 viewports (one per unique viewport-size bucket, deduplicating devices that share dimensions).

## Interaction Model

### Reference Device Selection

- Click any viewport container to make it the **reference device** (highlighted with a colored border)
- The reference viewport becomes the drag workspace
- All other viewports are read-only previews

### Element Selection

Click inside the reference iframe to select a VGP element. The tool accesses the iframe's DOM via `iframe.contentDocument` (same-origin).

**Draggable elements** (layout groups — moving these repositions all their children):

| Element | Description |
|---------|-------------|
| `.vgp-dpad` | D-pad container (4 directional buttons) |
| `.vgp-stick-zone` | Analog stick zone + knob |
| `.vgp-cbuttons` | C-button diamond (4 buttons) |
| `.vgp-ab` | A/B button group |
| `.vgp-shoulders` | Shoulder bar (L, R, spacer) — portrait only |
| `.vgp-z-portrait` | Z button wrapper — portrait only |
| `.vgp-start-portrait` | Start button — portrait only |
| `.vgp-start-landscape` | Start + Z wrapper — landscape only |

**Not draggable** (structural containers — moving these would break the flex/grid layout):

| Element | Reason |
|---------|--------|
| `.vgp-left` | Left panel — its position is fixed by CSS (`left: var(--offset-l)`). Moving the panel would break game canvas sizing. |
| `.vgp-right` | Right panel — same as above. |
| `.vgp-spacer` | Invisible flex spacer. No visual element to grab. |
| Individual buttons (`.vgp-a`, `.vgp-du`, etc.) | Positioned relative to their parent group. Moving one button out of its group is not a valid layout operation. |

Selected element gets a highlight outline and the inspector panel updates.

### Drag to Reposition

1. **Mouse down** on a draggable element — capture the element's primary `vgp-` class and current `getBoundingClientRect()`
2. **Mouse move** — apply a CSS `translate()` to the element. Show a tooltip with the delta: `dx: +12, dy: -8`
3. **Mouse up** — record the delta to the change log. Propagate to all other iframes:
   - Find the same element by class in each iframe's `contentDocument`
   - Apply the same pixel delta as a visual-only `translate()` transform
   - This is a **preview** — no CSS values are modified

### Keyboard Nudge

- Arrow keys move the selected element by 1px (fine-grained positioning)
- Shift+Arrow moves by 10px
- Escape deselects the current element
- Tab cycles through draggable elements in the reference viewport

### Change Propagation (Approach C)

Dragging on the reference device propagates to all other viewports as visual-only transforms. This lets the developer see the ripple effect of a position change across all device sizes. The previews are approximate (same pixel delta on different viewport sizes) — the actual responsive CSS translation is done by Claude when reading the export.

### Undo

- `Ctrl+Z` / `Cmd+Z` removes the last drag operation
- Per-element undo stack
- Undoing a change also removes the preview transforms from all viewports

## Inspector Panel

A fixed panel at the bottom of the page showing:

1. **Selected element** — class name, which container it's in, and its flex/grid role
2. **Computed CSS** — position, width, height, margin, padding, transform, flex properties
3. **Change log** — list of all drag operations with per-item undo buttons
4. **Export controls**:
   - "Copy JSON" — copies change log to clipboard
   - "Save to file" — downloads as `vgp-design-changes.json`
   - "Reset All" — clears all preview transforms across all viewports

## Workflow

1. Create symlink if not exists: `ln -sf web/static static` (from repo root)
2. Start local server: `python3 -m http.server 18888` (from repo root)
3. Open `http://localhost:18888/tests/design-mode.html` on desktop
4. Browse viewports, click one to set as reference
5. Click elements, drag to reposition (or arrow-key nudge), see all devices update
6. When satisfied, click "Copy JSON" or "Save to file"
7. Paste JSON into Claude conversation (or Claude reads the file)
8. Claude translates pixel deltas into responsive CSS changes (clamp/svh/custom properties)
9. Refresh design-mode page — this clears all preview transforms and loads the fresh CSS, showing the actual result across all devices
10. Repeat until layout is correct

## Constraints

- **No resize in v1** — only drag-to-reposition. Resizing requires translating pixel changes to `clamp()` values, which is a harder problem.
- **No server component** — pure static HTML/JS. CSS generation happens in conversation with Claude.
- **Same-origin only** — iframes must be served from the same origin as the design-mode page.
- **Browser chrome with-only** — design tool shows only the with-chrome variant per orientation. Full-viewport variants remain in the Playwright screenshot test.
